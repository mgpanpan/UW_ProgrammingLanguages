Submission from: Student 1

On this page, you will submit Homework 6 for peer assessment and assess submissions from some other students. Please follow the instructions carefully.

For submission, follow these instructions (same as for previous homeworks):

  • Click the "Attach a file" button below to upload your solutions file (just your solutions file, not your testing file). This should, somewhat strangely, put the name of your file in the text box above the "Attach a file" button. This is correct behavior. Do not paste file contents into the text box as this is unlikely to preserve indentation well.
  • Submit (or Re-submit) your assignment by clicking the big submit (or re-submit) button (after checking the Honor Code box).
  • You should see, "Your work was submitted. Review your work to make sure everything looks OK." Click on, "review your work."
  • You should see a box with the name of your file. Now you can click on this file name to get your solutions. They should appear formatted and indented exactly like the file you uploaded. Depending on your browser, you may need to save the file and view it in a separate editor.

For assessing others' submissions, first carefully read the instructions on the main peer assessment page. Then read the specific instructions for grading each problem. Because the file you are downloading contains a program you did not write, follow general security precautions: Do not run the code without checking that it does no harmful things. For completing peer assessment, you probably do not need to run the code at all since you are evaluating how the code is written.

Remember to do at least as good a job as you hope others will do for your assignment.

Overall evaluation/feedback

General instructions: Overall, this assignment has much more flexibility than earlier ones, so we will, as usual, present a sample solution, but we will also discuss some alternatives. We may not present all reasonable alternatives. Use your best judgment in evaluating the length, OOP-style, and general Ruby-style of a solution. Overall, use of super is a good thing, as is use of helper methods.

Remember this general description of the numeric scores:

5: This is a flawless or nearly flawless solution. Everything in the code is there for a reason, it is easy to read, it is indented well, and it uses the appropriate language constructs from the course. It is hard to imagine any correct solution being much simpler.

4: The code is very good style, but there are one or two places where there was a clearly better thing to do. For example, maybe the student wrote "if e then true else false." Or maybe there is an unnecessary case or a redundant computation. Or the code is mostly nicely formatted, but one part was formatted badly. You should be able to justify a 4 instead of a 5 by giving a specific place or two where there is something that is not as good as it could be.

3: The code is okay, but it is too difficult to read in some places because there are simpler ways to say the same thing. Or maybe the student did not follow the instructions completely in terms of what language constructs to use or not use, but not in a way that made a huge difference. A good way to think about a "3" score is that it is a "4" score where the list of issues is too long.

2: This code might be a correct solution, but it was very difficult to read. There were many style problems. The overall structure is not clear. Or the solution is far more complicated or longer (e.g., 2-3 times as long) as it needs to be.

1: This solution is too far away from a correct one to be reasonably graded. Most or all cases are not handled correctly. The homework was not really completed so the code for this problem is too short to be evaluated.
